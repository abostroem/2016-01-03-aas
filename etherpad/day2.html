Overview of Use: edit/create files, stage files, commit files to repository

Status:

    # new files dropped into git repository will show up when you type:
        git status 
    # git status is useful for seeing modified files, new files, staged files 
    # git status gets cleared when files are committed. 

Tracking/Adding to Staged Area:

    # to track a file use (file gets moved to the staging area):
        git add filename 

Committing Files to Repository:

    # to commit a file with a message (files gets moved to repository): 
        git commit -m "commit message" 
    # make commit messages unique and informative, think: "message to my future self, and my past self won't return emails"

General Utility Functions:

    # in general, more options available from help pages (even to specific sub git commands): 
        git help diff 
        git help log 
        

LOG

    # history of git commits (default is reverse chronological): 
        git log 
    # will show commit ID hash (long string of numers and letters), unique identifier for individual commits
    # use to reference specific commits 
    # git status will also show changed files in the repository as 'modified'

DIFF

    # to see the difference in file use: 
        git diff filename 
    # track the changed file, then commit when ready
    # getting diffs of staged files: 
        git diff --staged 
    # getting diffs of versions
    # current version in repository is called HEAD 
    # pervious version in repository is called HEAD~1 
    # etc. 
    # so to compare current and previous versions in repository: 
        git diff HEAD~1 mars.txt
    # can also go two previous back
  •     git diff HEAD~2
  • # Only look at last 3 commit messages
    • git log -3 

CHECKOUT

    # use to discard working directory changes and/or recover previous version, 
    # e.g., mars.txt was overwritten and you want previous committed version: 
        git checkout HEAD mars.txt 
    # ALWAYS put file name otherwise the entire repository can be recovered with some detached head (!?!) 
    #Shorthand for revert to previous version
  •     git checkout -- jupiter.txt

Python 3 formatting:
Python 2 used %s, %f, ... for formatting: e.g. print "Invalid numeric value (%s) for minutes" % sa[1]
Python 3 uses '{}'.format() for formatting: e.g. print("Invalid numeric value ({}) for minutes".format(sa[1]))
  • Examples: https://docs.python.org/3.1/library/string.html#format-examples

When you type python make_power.py from the shell, make_power.py is executed line by line. So it does the following:
  1. import sys
  2. define all of our functions
  3. run what is under if __name__ == __main__

For code you want to run on one file, you can run either from the command line or in a GUI like the notebook. When you want to automate, usually more useful to run as a script (easier to run on multiple files, more reproducible, easier to version control)

sys.argv returns a list with the the filename as its first argument and all of the command line arguments as other elements in the list.

More on if __name__ == ""__main__":
Details on __main__: https://docs.python.org/3/library/__main__.html
When you run from the command line (shell) as a script __name__ == "__main__" so everything inside the if __name__ == "__main__" statement is run. When you import your module, __name__  is equal to the name of your module, so everything under the if statement is not executed.

This is useful - when you run a script from the command line there is generally a set analysis you want to do, when you import a script you usually want to use one or more of the functions and not execute the predefined code.
